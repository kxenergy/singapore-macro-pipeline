## What This Project Is

An end-to-end data engineering pipeline that ingests Singapore macroeconomic data from three government APIs (SingStat, MAS, Enterprise SG), transforms it through a Bronze→Silver→Gold medallion architecture using dbt + DuckDB, and serves it as a live three-page Evidence.dev dashboard at [sgmedallion.netlify.app](https://sgmedallion.netlify.app).

## Prerequisites

```bash
python --version    # 3.10+
dbt --version       # 1.11+
node --version      # 18+ (for dashboard)
```

Install Python dependencies:
```bash
pip install dbt-core dbt-duckdb duckdb requests pandas
```

Configure dbt by creating `~/.dbt/profiles.yml` (this file is gitignored — never commit it):
```yaml
sg_macro_dbt:
  target: dev
  outputs:
    dev:
      type: duckdb
      path: /absolute/path/to/singapore-macro-pipeline/data/warehouse/sg_macro.duckdb
      threads: 4
```

## Running the Pipeline

```bash
# Step 1 — Ingest (run from project root or ingestion/)
python ingestion/ingest_cpi.py
python ingestion/ingest_mas_fx.py
python ingestion/ingest_trade.py

# Step 2 — Transform and test
cd sg_macro_dbt
dbt run
dbt test

# Step 3 — Export Gold mart to CSV for dashboard
python ingestion/export_csv.py

# Step 4 — Load dashboard sources and view
cd dashboard
npm install          # first time only
npm run sources
npm run dev          # http://localhost:3000
```

## dbt Commands

```bash
cd sg_macro_dbt

dbt run                        # run all models
dbt run --select staging       # run only Silver staging models
dbt run --select marts         # run only Gold mart models
dbt test                       # run all 15 data quality tests
dbt test --select fct_sg_macro_monthly  # test one model
dbt docs generate && dbt docs serve     # lineage graph at http://localhost:8080
dbt clean                      # remove target/ and dbt_packages/
```

## Architecture

### Data Flow

```
Government APIs → Python ingestion → DuckDB Bronze → dbt staging (Silver) → dbt marts (Gold) → CSV export → Evidence.dev dashboard
```

### Medallion Layers in DuckDB (`data/warehouse/sg_macro.duckdb`)

| Schema | Tables | Role |
|---|---|---|
| `main` (Bronze) | `bronze_cpi_raw`, `bronze_fx_raw`, `bronze_trade_raw` | Raw, immutable, timestamped |
| `main_staging` (Silver) | `stg_cpi`, `stg_fx`, `stg_trade` | Typed, renamed, unpivoted; materialized as views |
| `main_marts` (Gold) | `fct_sg_macro_monthly`, `dim_calendar` | Joined analytical mart; materialized as tables |

### dbt Model Structure (`sg_macro_dbt/models/`)

- `staging/sources.yml` — declares Bronze tables as dbt sources
- `staging/schema.yml` — column-level tests and descriptions for Silver models
- `staging/stg_cpi.sql` — parses dates, casts types, computes MoM/YoY via window functions
- `staging/stg_fx.sql` — filters to 7 key currencies, adds inverse rate
- `staging/stg_trade.sql` — unpivots wide→long, filters to total imports/exports
- `marts/dim_calendar.sql` — date dimension (quarter, season, month name)
- `marts/fct_sg_macro_monthly.sql` — central fact table joining all three Silver models
- `marts/schema.yml` — 15 automated data quality tests on the Gold mart

### Dashboard (`dashboard/`)

Three pages in `dashboard/pages/`:
- `index.md` — Executive Summary (CFO audience)
- `analysis.md` — Full Signal Analysis (analyst audience)
- `pipeline.md` — Pipeline Architecture (recruiter/non-technical audience)

Data source: `dashboard/sources/sg_macro/macro_monthly.csv` (generated by `export_csv.py`). The dashboard uses the CSV connector (`connection.yaml` declares `type: csv`) because Evidence.dev's DuckDB connector has compatibility issues with Node v24.

### Ingestion Scripts (`ingestion/`)

All three scripts follow the same ETL pattern: `extract()` → `transform()` → `load()`. Key behaviors:
- **Idempotent**: delete-before-insert on today's date prevents duplicate rows on re-runs
- **Logged**: structured logs written to `logs/ingest_*.log` (gitignored)
- **Paths**: use `os.path` relative to `BASE_DIR` (parent of `ingestion/`) — portable across machines

### CI/CD (`.github/workflows/pipeline.yml`)

Runs `dbt run && dbt test` on the 1st of each month at 9am UTC, and on every push. The workflow does **not** run the Python ingestion scripts — those require live API access and run locally or are triggered manually via `workflow_dispatch`.

## Key Design Decisions

- **DuckDB over cloud warehouse**: same dbt models run against Snowflake/BigQuery by changing `profiles.yml` only — no SQL rewrites needed.
- **CSV export workaround**: `export_csv.py` must be run before `npm run sources` whenever the Gold mart is updated. This is a known pragmatic workaround for Node v24/DuckDB connector incompatibility.
- **`data/warehouse/` is gitignored**: the DuckDB file is never committed. Always run the full pipeline after cloning.
- **`profiles.yml` is gitignored**: each developer maintains their own with the correct absolute path to the `.duckdb` file.
